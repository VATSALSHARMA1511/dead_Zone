DEADZONE — Full Project Context
What It Is
Top-down neon tactical zombie survival shooter built in Python + Pygame. Production-style project with AI integration, deployed backend, and analytics. Built for portfolio/resume purposes targeting product company placements.

Tech Stack

Game: Python, Pygame
Backend: FastAPI, PostgreSQL, deployed on Railway
AI: Groq API (llama-3.3-70b-versatile)
RAG: ChromaDB + sentence-transformers (all-MiniLM-L6-v2)
Deployment: Railway (auto-deploy from GitHub)
GitHub: github.com/VATSALSHARMA1511/dead_Zone
Live URL: https://deadzone-production-759b.up.railway.app

Every System Built
1. Core Game Systems

Top-down shooter, WASD movement, mouse aim, LMB shoot
3 weapons: pistol, shotgun, SMG (1/2/3 or scroll to switch)
Sprint (Shift), Dash with iframes (Space), stamina system
HP-based movement penalties — slow at low health
HP-based ammo drain — burns faster when hurt
Streak multiplier — kill chain boosts score
Screen shake on shoot, muzzle flash particles, blood particles
Survival score — +wave_number points per second alive

2. Procedural Sector World

3x3 grid of procedurally assigned sectors
4 sector types: AI_CORE, QUARANTINE, CONTAINMENT, REACTOR
Each has unique color palette, grid style, decals, accent border
Pre-rendered to surface at game start for performance
Tactical obstacles with collision — 8 fixed positions

3. Player System (player.py)

Custom sprite support — uploadable photo, circular crop, rotation
Dash trail visual, bob animation while sprinting
Hit flash on damage, iframes after hit
Gun barrel drawn on top of sprite always
Fallback circle rendering if no photo uploaded

4. Zombie System (zombie.py)

4 types: normal, fast, tank, boss
Custom sprite support per type
Type ring indicators
Smooth interpolation movement toward player
8% speed increase per wave
Attack rate system per type

5. AI Wave Director (waves.py)

Groq API called in background thread at END of each wave (during cooldown)
Full wave history (last 5 waves) sent as context — kills, damage, accuracy, composition
Wave-anchored zombie floor: min = max(6, 4 + next_wave * 2)
Response fully validated and ratio-normalized before use
Boss waves deterministic — every 5 waves, AI has no say
Falls back to default composition if API fails or times out
AI message displayed on HUD for 5 seconds per wave

6. RAG Persistent Memory (rag_director.py)

Full RAG pipeline: Embed → Store → Retrieve → Augment → Generate
Each completed run converted to natural language summary
Embedded using sentence-transformers (all-MiniLM-L6-v2), stored in ChromaDB
Before each wave composition request, ChromaDB queried for similar past runs
Top 3 similar runs injected into Groq prompt as additional context
Persists to ./deadzone_memory/ folder locally across sessions
Module-level singleton via rag_director.get()
Fully graceful — if ChromaDB unavailable, game works normally

7. Collision System (collision.py)

Centralised — all collision in one system, no circular imports
Bullets vs zombies — returns hit count for accuracy tracking
Player vs zombies — damage + iframes check
Zombie separation — push-apart to prevent stacking
Player vs obstacles — AABB resolution
Zombie vs obstacles — AABB resolution with 0.7 damping
Player + zombie world bounds clamping

8. Accuracy Tracking

_bullets_fired incremented when player shoots
_bullets_hit incremented from collision system return value every frame
waves.stat_bullets_hit also updated for per-wave accuracy
Accuracy calculated at game over: (hits / max(1, fired)) * 100
Sent to Groq for game over analysis and wave director

9. HUD System (hud.py)

Bottom panel: HP bar (color coded), stamina bar, dash cooldown arc
Top bar: score, wave number, FPS, particle count, streak display
Wave alert: countdown + "WAVE X INCOMING" / "⚠ BOSS WAVE" during cooldown
Wave clear banner: "✓ WAVE CLEARED" fades in/out
Weapon slots: 3 slots with ammo pips, active weapon highlighted
Crosshair: custom drawn, follows mouse
Reload indicator: arc around crosshair showing reload progress
Wave summary panel: shown during cooldown, displays kills/accuracy/damage with letter grade S/A/B/C/D

10. Game Over Screen (menus.py — GameOverScreen)

Shows score, wave, kills, best score
AI tactical analysis panel — threaded Groq call on game over
"ANALYSING RUN..." animated dots while waiting
Analysis fades in when ready
Grade system based on performance

11. Player Name Entry (menus.py — NameEntryScreen)

Text input screen before sprite picker
Max 12 characters, uppercase forced
Stores in player_name_store.py module singleton
Used on leaderboard instead of hardcoded "PLAYER"

12. Sprite Picker (menus.py — SpritePickerScreen)

Upload player photo and zombie photo via native file dialog
Circular preview with checkmark badge
Clear button per slot
Skippable — defaults used if no photo chosen

13. Backend (server.py)
Endpoints:

POST /score — submit run with name, score, wave, kills, ai_review
GET /scores — top 10 leaderboard JSON
GET / — leaderboard HTML page
GET /analytics — analytics dashboard HTML page
GET /analytics/summary — total runs, unique players, avg/best wave, best score, avg kills
GET /analytics/wave-distribution — wave death distribution
GET /analytics/accuracy-trend — last 20 runs for trend chart
GET /analytics/top-players — per-player aggregated stats

DB schema:
sqlCREATE TABLE scores (

    name      TEXT    NOT NULL,
    score     INTEGER NOT NULL,
    wave      INTEGER NOT NULL,
    kills     INTEGER NOT NULL,

14. Score Posting Flow (game.py)

_delayed_post thread waits up to 6 seconds for AI analysis to complete
Posts score + AI review together in one request
After posting, calls rag_director.get().store_run() to persist run to vector store
Uses player_name_store.name for player name

15. Analytics Dashboard (/analytics)

Summary stat cards: total runs, unique players, avg wave, best wave, best score, avg kills
Wave distribution bar chart — color coded, shows where players die most
Score trend line chart — last 20 runs plotted on canvas
Player stats table — best score, avg score, best wave, avg wave, total kills, run count
Nav links between leaderboard and analytics

16. Leaderboard Page (/)

Top 10 scores, auto-refreshes every 10 seconds
AI review shown under each score in italic
Gold/silver/bronze rank labels
Nav link to analytics

Key Architecture Decisions

All collision in one system — no circular imports, easy to optimize
AI called in daemon threads — never blocks game loop
RAG singleton — initialized once, shared across waves.py and game.py
player_name_store.py module singleton — avoids passing name through constructors
Boss waves deterministic — AI cannot override, always wave % 5 == 0
Wave composition validated and normalized — model output never trusted blindly
_delayed_post pattern — waits for async AI analysis before posting score
